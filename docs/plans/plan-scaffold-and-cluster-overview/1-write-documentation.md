# Step 1: Write documentation

Write all documentation up front, using the plan (`docs/plans/new/plan-scaffold-and-cluster-overview.md`, section 1, substeps 1-7) as the source of truth. This is documentation-first: the docs describe the feature and conventions as they are intended to work, so the user can read and understand what they are getting before any code is written. The final step revisits every doc to reconcile it against the implementation.

## Files to create

1. **`readme.md`** (repo root): one-paragraph project description (local-only Kubernetes dashboard wrapping `kubectl`); "Requirements" (`kubectl` on `PATH`, `bun` via mise, at least one configured kubeconfig context); "Getting started" (`mise install`, `bun install` in `backend/` and `frontend/`, `bun run dev` in each); "Documentation" linking the six guide files under `docs/`: `architecture.md`, `api.md`, `e2e-testing.md`, `user-guide.md`, `audit-log.md`, `roadmap.md`.

2. **`claude.md`** (repo root): concise project-level guidance covering purpose/stack/repo-layout/doc-map; file naming (kebab-case files, PascalCase React identifiers); source layout (everything under `src/`, backend tests mirror the source tree under `backend/src/tests/`, not co-located); code style (idiomatic TS casing, named exports only, 4-space indent, one statement per line, always-braces, Allman-style `else`); module style (ESM only, never `require`, never dynamic `import`); HTTP (axios via `api-client.ts`); data fetching (React Query everywhere); shared state (`KubeContextProvider`/`useKubeContext()`); tables (`@tanstack/react-table` + MUI); routing (React Router 7, routes in `app.tsx`, pages vs components); icons (Font Awesome, registered in `lib/font-awesome.ts`); backend runtime rules (no Bun-specific APIs, no `*Sync` APIs, async everywhere); testing discipline (every backend non-React module tested under `backend/src/tests/`, `index.ts` exempt, entire frontend including `lib/*.ts` not unit-tested, never `test.skip`/`describe.skip`, always `describe`/`test` never `it`, no fudged assertions); run instructions (backend always launched with cwd=`backend/` so `"./logs"` resolves to `backend/logs/`); deployment (none, binds 127.0.0.1, no CORS); kubectl assumption; **read-only kubectl invariant** (create/write/edit commands must never be added to the adapter); audit log rule (every kubectl call audited before spawn via the private `kubectl(args)` helper).

3. **`docs/architecture.md`**: repeat the architecture diagram (the one including `audit-log.ts` and the private `kubectl(args)` audit-then-run path, see plan Architecture section); explain each layer; document 127.0.0.1-only bind; document the type-mirroring choice; document how kubectl failures surface (adapter throws plain `Error` with kubectl stderr as message; route returns HTTP 500 JSON). Include a **"Local-only threat model / accepted risks"** subsection covering (a) kubectl stderr returned verbatim (issue 24) and (b) no Host-header / DNS-rebinding guard (issue 25). Document the read-only kubectl invariant and the audit log integration (cwd-relative `"./logs"` resolving to `backend/logs/`).

4. **`docs/api.md`**: one section per endpoint with request/response types, status codes, curl examples. Endpoints: `GET /api/contexts`, `POST /api/contexts/current` (200 / two 400 messages: empty-or-non-string name and leading-`-` name / 500 on kubectl failure), `GET /api/cluster/overview`, `GET /api/cluster/nodes`. Note routes are local-only and unauthenticated and that kubectl stderr is surfaced verbatim in 500 bodies (accepted tradeoff, issue 24).

5. **`docs/e2e-testing.md`**: user-facing end-to-end test guide. Prerequisites (including `jq`/`curl`/`bash` the smoke script needs, issue 26), start-the-stack, smoke checks, interaction checks, backend-only curl checks, triage list. Companion to `scripts/smoke.sh`. The interaction checks **must include two explicit named manual steps** standing in for the frontend's missing unit coverage (issue 22): **Context-switch refetch** (switch context, confirm both tiles and nodes table refetch) and **Unset-current-context gating** (`kubectl config unset current-context`, confirm "Select a context…" message and that the nodes table fires no request).

6. **`docs/user-guide.md`**: end-user usage guide. What Karse is (and isn't); prerequisites; setup; tour of the cluster home page (header, stat tiles, nodes table, status colours, age computation, sort indicators, search); switching contexts; **audit log** (brief: every kubectl call logged under `backend/logs/<YYYY>/<MM>/<DD>/<HH>.log` in local time, retained 3 months, pointer to `docs/audit-log.md`); limitations; troubleshooting; pointers to roadmap/architecture/api.

7. **`docs/roadmap.md`** and **`docs/audit-log.md`**:
   - `roadmap.md`: short intro, then a flat numbered list of at least ten upcoming features (one sentence each), including a **"Host-header allowlist / DNS-rebinding guard"** hardening entry (issue 25). End with an "Already shipped" section listing the cluster overview + nodes view delivered by this plan.
   - `audit-log.md`: title "Karse audit log"; what gets logged (every kubectl invocation, appended to a rolling text file); where (`backend/logs/<YYYY>/<MM>/<DD>/<HH>.log` from local time; cwd-relative `"./logs"` resolving to `backend/logs/` because the backend always runs with cwd=`backend/`); format (one line per call, local-time ISO 8601 with explicit offset, never `Z`); retention (dated dirs older than 3 months deleted at startup, running-process logs not pruned mid-run); how to read (standard Unix text tools); read-only kubectl by design.
