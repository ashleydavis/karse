# guided-commands

## Overview

Two display-only surfaces for kubectl commands:

- A **Commands** tab on each resource detail page (pod, node, deployment, stateful set, daemon set) that lists suggested kubectl commands for *acting on* that resource.
- **Page help**: a question-mark button in the header, on every page backed by cluster data, opening a panel that says where the current page's data came from and lists the read-only kubectl commands that reproduce it.

The commands on both surfaces are pure display strings the user copies and runs themselves. Nothing in this feature executes commands or wires them to any adapter, so it does not violate the read-only invariant (see `read-only-invariant`).

Backed by: `frontend/src/lib/guided-commands.ts`, `frontend/src/lib/page-help.ts`, `frontend/src/components/commands-tab.tsx`, `frontend/src/components/page-help.tsx`, `frontend/src/components/command-row.tsx`.

## Behaviour: the Commands tab

- `buildGuidedCommands(target)` returns a list of `{ label, command }` for the target resource. It is a pure function: no side effects, no execution.
- Pod commands: describe, view logs, follow logs, open a shell, get YAML, delete. A namespace, when present, is appended as `-n <namespace>`.
- Node commands: describe, get YAML, list pods on node, cordon, drain (`--ignore-daemonsets --delete-emptydir-data`), uncordon.
- Workload commands (deployment/statefulset/daemonset): describe, get YAML, restart rollout, rollout status, and delete; deployments and stateful sets also get a scale-replicas suggestion.
- The Commands tab renders each suggestion as a labelled code block with a copy-to-clipboard button, and a prominent read-only note: "Karse is read-only and never runs these commands. Copy and run them yourself."
- Commands word-wrap so the full command is always visible without horizontal scrolling, and the tab uses the full detail-page width.
- A search box filters the command list (`filterGuidedCommands(commands, query)`), fuzzy-matching the query against each command's label and command text. An empty query shows the full list.
- Some suggestions are mutating kubectl commands (delete, scale, drain, cordon). They are text only; Karse never runs them.

## Behaviour: page help (where this data comes from)

- A question-mark button sits in the header on every page, next to the colour-mode button, so the help is reachable from anywhere without cluttering the page. The panel it opens is closed by default.
- `buildPageHelp(pathname, { context, namespace })` returns `{ title, source, commands }` for the page at that pathname, or `null` for a page with no cluster data behind it (About, Config) — the header then renders no help button at all. It is a pure function: it derives everything from the route and the active selection, and executes nothing.
- `source` is a plain-English statement of where the page's data came from, e.g. that the Errors feed is derived from Warning events plus the pods, or that a container has no object of its own and its page comes from the pod.
- `commands` are the read-only kubectl queries Karse itself runs through the adapter to build that page, rewritten as commands the user can paste into a terminal. They are pinned to the selected context (`--context <name>`, omitted when no context is selected) and scoped to the selected namespace (`-n <namespace>`, or `-A` when no namespace is selected), so they return the same data the page is showing.
- A detail page's commands use the resource's own namespace from the route, not the selected namespace.
- The panel shows the page title, the data-source statement, and each command as a labelled code block with a copy button (the same `CommandRow` the Commands tab uses), under a note that these are the read-only commands behind the page and Karse only shows them.
- Page help lists only read-only queries. Commands that act on a resource stay on the Commands tab.

## Acceptance Criteria

- [x] Command suggestions are generated by a pure function with no execution.
- [x] Pod, node, and workload (deployment/statefulset/daemonset) targets each have a tailored command list.
- [x] A namespace, when present, is appended as `-n <namespace>`.
- [x] The Commands tab renders each command with a copy button and a read-only disclaimer.
- [x] Commands word-wrap (no horizontal scroll) and the tab uses the full detail-page width.
- [x] The command list is searchable via a filter box.
- [x] Mutating suggestions are shown as copyable text only and are never executed by Karse.
- [x] Every page backed by cluster data exposes a help affordance in the header that states where that page's data came from.
- [x] The help shows the concrete read-only kubectl command(s) the user can run to get the same information themselves, for that page's context.
- [x] The commands are correct for the page and reflect the selected context and namespace.
- [x] The help is discoverable but unobtrusive: one header button, panel closed by default, and no button at all on a page with no cluster data behind it.
